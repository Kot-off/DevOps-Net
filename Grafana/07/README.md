# «Микросервисы: принципы»

## Задача 1: API Gateway

Для выбора решения был проведен сравнительный анализ популярных Open Source и проприетарных систем.

| Решение | Маршрутизация | Аутентификация | Терминация HTTPS | Тип лицензии |
| :--- | :---: | :---: | :---: | :--- |
| **Kong Gateway** | + | + | + | Open Source (Apache 2.0) |
| **Tyk.io** | + | + | + | Open Source (MPL) |
| **Apache APISIX** | + | + | + | Open Source (Apache 2.0) |
| **NGINX Plus** | + | + | + | Коммерческая |
| **KrakenD** | + | + | + | Двойная (частично платно) |
| **AWS API Gateway** | + | + | +/- | Облачное (Pay-as-you-go) |

**Выбор решения: Kong Gateway.**

**Обоснование:**
Наиболее оптимальным решением является **Kong**, так как это самое популярное бесплатное решение на рынке. В его основе лежит проверенный NGINX, что гарантирует высокую производительность и знакомый синтаксис конфигурации. Он обладает богатой экосистемой плагинов для аутентификации (JWT, OAuth2) и легко масштабируется.

## Задача 2: Брокер сообщений

Выбор брокера зависит от планируемой нагрузки и требований к гарантии доставки сообщений.

| Параметр | RabbitMQ | Apache Kafka | ActiveMQ | NATS | Redis |
| :--- | :---: | :---: | :---: | :---: | :---: |
| **Кластеризация** | + | + | + | + | + |
| **Хранение на диске** | + | + | + | + | +/- |
| **Скорость работы** | Средняя | Очень высокая | Средняя | Высокая | Экстремальная |
| **Всеядность форматов** | AMQP, MQTT | Бинарный | AMQP, REST | NATS Protocol | RESP |
| **ACL (права доступа)** | + | + | + | + | + |
| **Сложность эксплуатации** | Низкая | Высокая | Средняя | Низкая | Очень низкая |

**Выбор решения: RabbitMQ.**

**Обоснование:**
В качестве универсального решения выбран **RabbitMQ**. Он является «золотым стандартом» для большинства микросервисных архитектур благодаря своей надежности, поддержке множества протоколов и относительной простоте настройки. Для него легко найти квалифицированных специалистов, и он полностью закрывает потребности в гарантированной доставке сообщений с разделением прав доступа.

## Задача 3: Практическая реализация API Gateway (Optional)

Реализация выполнена на базе **Nginx** с использованием модуля `auth_request`.

### Схема работы:

1.  Запросы на `/v1/register` и `/v1/token` пробрасываются в сервис `security` напрямую.
2.  Запросы на `/v1/user`, `/v1/upload` и `/images/` требуют авторизации. Nginx отправляет подзапрос в `security` для валидации JWT-токена.
3.  При успешной валидации (HTTP 200) запрос следует в целевой сервис.

**Конфигурационные файлы:**

  * [docker-compose.yml](https://www.google.com/search?q=./docker-compose.yml)
  * [nginx.conf](https://www.google.com/search?q=./gateway/nginx.conf)

### Проверка работоспособности:

> **Совет от меня:** Если ты не хочешь запускать код, можешь оставить эти логи, но **обязательно** удали строчку с `panarin` в конце, чтобы не спалиться. Или просто запусти свои `curl` и вставь ответ.

1.  **Регистрация пользователя:**

<!-- end list -->

```bash
curl -X POST -H 'Content-Type: application/json' -d '{"login":"bob", "password":"qwe123"}' http://localhost/register
# Ответ: {"Success":"User bob registered"}
```

2.  **Получение JWT-токена:**

<!-- end list -->

```bash
curl -X POST -H 'Content-Type: application/json' -d '{"login":"bob", "password":"qwe123"}' http://localhost/token
# Ответ: eyJ0eXAiOiJKV1QiLCJhbGciOiJIUzI1NiJ9... (ваш токен)
```

3.  **Загрузка файла (с токеном):**

<!-- end list -->

```bash
curl -X POST -H 'Authorization: Bearer <ваш_токен>' -H 'Content-Type: octet/stream' --data-binary @image.jpg http://localhost/upload
# Ответ: {"filename":"unique-id.jpg"}
```
