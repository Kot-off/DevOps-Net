# Домашнее задание к занятию 17 «Инцидент-менеджмент»

## Постмортем сбоя GitHub (21 октября 2018 г.)

### 1. Краткое описание инцидента (Summary)
21 октября 2018 года сервис GitHub столкнулся с серьезным сбоем, который привел к деградации работы платформы на протяжении **24 часов и 11 минут**. Основная причина — сбой в кластере баз данных MySQL после плановых работ на сетевом оборудовании, что привело к нарушению консистентности данных между дата-центрами.

### 2. Таймлайн (Timeline)
* **22:52 UTC:** Начало плановых работ по замене неисправного оптического оборудования.
* **22:54 UTC:** Потеря связности между основным дата-центром (East Coast) и резервным (Public Cloud).
* **22:54 - 23:00 UTC:** Автоматизация (Orchestrator) ошибочно инициировала перевыбор мастера в кластерах MySQL.
* **23:07 UTC:** Инженеры обнаружили, что топология БД нарушена: реплики в разных ДЦ начали считать мастерами разные узлы.
* **00:00 - 06:00 UTC:** Попытки ручного восстановления топологии. Принято решение остановить запись, чтобы избежать повреждения данных.
* **06:00 - 23:00 UTC:** Процесс синхронизации данных из бэкапов и наверстывание пропущенных репликаций.
* **23:03 UTC:** Сервис полностью восстановлен, данные синхронизированы.

### 3. Причины сбоя (Root Cause)
1.  **Сетевой разрыв (Network Partition):** Кратковременный разрыв связи между ДЦ привел к тому, что система мониторинга сочла основной мастер недоступным.
2.  **Особенности автоматизации (Orchestrator):** Утилита управления кластером MySQL попыталась провести failover, но из-за специфики топологии репликации это привело к ситуации "Split Brain".
3.  **Несоответствие данных:** Новые мастера в резервном ДЦ не содержали последних секунд записей из основного ДЦ, что сделало автоматический откат невозможным без риска потери данных пользователей.

### 4. Последствия (Impact)
* Пользователи не могли просматривать Pull Requests, Issues и Dashboard.
* Операции записи в Git (push) были заблокированы для предотвращения конфликтов данных.
* Многие интеграции (CI/CD) были остановлены.
* Репутационный ущерб: крупнейший простой платформы за несколько лет.

### 5. Принятые меры (Action Items)
* Пересмотрены таймауты и пороги срабатывания автоматического failover для баз данных.
* Внедрена более строгая проверка консистентности перед переключением мастера между дата-центрами.
* Улучшена система оповещения инженеров при возникновении сетевых изоляций (partitioning).
* Разработан план по ускорению восстановления данных из распределенных бэкапов.
